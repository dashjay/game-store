## 2. 端到端吞吐（真实 redis-py 经 TCP，单连接）

复跑：

```bash
cargo run -p gamestore-datanode --release -- --port 6392 --data-dir /tmp/gs-bench
python3 tests/bench_throughput.py --port 6392 --ops 20000 --pipeline 100
```

| 工作负载 | 顺序（单命令往返） | pipeline(100) |
| --- | --- | --- |
| SET 64B | ~21,000 ops/s（47.5 µs/op） | ~81,000 ops/s |
| GET | ~25,500 ops/s（39.3 µs/op） | ~152,000 ops/s |
| HSET 玩家字段 | ~22,200 ops/s（45.1 µs/op） | ~73,500 ops/s |
| HGET 玩家字段 | ~24,700 ops/s（40.5 µs/op） | ~124,000 ops/s |
| ZADD 排行榜 | ~20,700 ops/s（48.3 µs/op） | ~71,300 ops/s |
| LPUSH 日志 | ~22,700 ops/s（44.0 µs/op） | ~85,200 ops/s |

顺序模式受客户端往返延迟（含 Python 客户端开销）支配（~40–48 µs/op，而服务端
执行只占其中 <10 µs）；pipeline 模式逼近单连接的服务端处理上限。

## 3. MR-0018 "引擎调用同步内联" 决策复核（I-07 议题）

MR-0018 决定 Phase-1 引擎调用在 tokio worker 上**同步内联**执行、不走
`spawn_blocking`，并约定 I-07 出数后复核。本基线支持维持该决策：

- 热路径点操作（GET/SET/HGET/HSET/ZADD/LPUSH）服务端执行 **0.4–7.4 µs**，
  与 `spawn_blocking` 一次任务移交 + 唤醒的开销（微秒级，且引入 blocking
  池排队抖动）同量级甚至更低——外移只会净增延迟。
- 最重的范围命令（`HGETALL` 50 字段 ~104 µs、`ZRANGE` 100 成员 ~18 µs）
  仍远低于 1 ms，不足以饿死同 worker 上的其它连接。
- 真正长耗时的 `COMPACT` 是测试/admin 动词，不在热路径（维持原判断）。

**结论：维持同步内联。** 触发重审的条件（记录在案）：出现 ≥ 毫秒级的热路径
命令（如未来的大 range/SCAN），或 I-08 WAL 落盘（fsync）进入前台写路径时，
按新形态重新评估（届时写路径本就要重构）。
