# 05 · 分片、路由与扩缩容

> 本文件说明 GameStore 如何把键空间切分到多个分片、如何路由请求、以及如何在不停服的前提下扩缩容。

## 1. 分片模型：兼容 Redis Cluster 的 16384 槽位

- 键空间被划分为 **16384 个哈希槽（slot）**，槽号 = `CRC16(key) % 16384`（与 Redis Cluster 一致）。
  使用 hash tag 时仅 `{...}` 内子串参与计算（见 [`01-workload-data-model.md`](01-workload-data-model.md) §3.2）。
- 多个连续/离散的槽位归属于一个 **分片（Shard）**；每个分片是一个 Raft 组（见 [`04-replication-consistency.md`](04-replication-consistency.md)）。
- **槽位 → 分片 → 副本位置** 的映射由 **PD** 维护，并带 **epoch（版本号）** 以便检测过期路由。

> 选择哈希槽而非范围分片的理由：负载天然均匀（玩家 ID 哈希后离散）、路由计算简单、与 Redis 生态/客户端一致。

## 2. 路由

### 2.1 经 Proxy 路由（默认）
1. Proxy 从 PD 同步 **路由表缓存**（槽位 → 分片 → 节点），并订阅增量更新。
2. 对每个请求计算槽位 → 查表得目标分片 Leader → 转发。
3. 若因迁移/选主导致路由过期，后端返回带新位置的重定向，Proxy **内部重试** 到正确节点，
   对客户端隐藏 `MOVED`/`ASK`。

### 2.2 智能 SDK 直连（可选）
- 支持 Redis Cluster 重定向语义的客户端可缓存槽位表、直连分片 Leader，省去 Proxy 一跳。
- 收到 `MOVED`（永久迁移）/`ASK`（迁移进行中）时更新本地路由并重试。

## 3. 扩容（加节点）

目标：吞吐/容量随节点线性增长，过程对业务透明、不丢数据、不长时间阻塞。

1. 新节点注册到 PD，加入集群。
2. PD 制定再均衡计划：把部分 **槽位/分片副本** 迁移到新节点，使负载更均匀。
3. 迁移以 **分片为单位**，两种基本手段：
   - **副本搬迁（transfer replica）：** 通过 Raft 成员变更，在新节点上加副本、追平数据后移除旧副本。
     数据搬运走"快照 + 增量日志"，**不阻塞前台读写**。
   - **分片分裂（split）：** 当单分片过大/过热，将其槽位集合一分为二，形成两个 Raft 组，分散到更多节点。
4. 迁移完成后 PD 更新槽位映射与 epoch，Proxy/SDK 增量感知新路由。

### 3.1 迁移期间的请求处理
- 迁移过程中，落在"正在搬迁槽位"的请求按 Redis Cluster 语义处理：
  尚未迁走 → 由原分片服务；正在迁移的 Key → 通过 `ASK` 重定向到目标分片。
- Proxy 对客户端屏蔽这一过程（内部重试），SDK 客户端按 `ASK`/`MOVED` 处理。

## 4. 缩容（减节点）

1. PD 标记待下线节点，将其上所有分片副本 **逐个搬迁** 到其他健康节点（保持每个分片始终多数派可用）。
2. 全部搬迁完成且复制因子满足后，安全移除该节点。
3. 全程不影响整体可用性。

## 5. 负载均衡与热点治理

- **数据/Leader 均衡：** PD 依据各节点的容量、QPS、Leader 数等指标，持续做 Leader 转移与副本搬迁，
  避免单节点过载。
- **跨 AZ 反亲和：** 同一分片的 3 副本强制分散到 3 个 AZ。
- **热点 Key：**
  - 监控识别热点（高 QPS 的单 Key/单槽）。
  - 缓解手段：Proxy 侧对热点读做短 TTL 本地缓存、对热点写做合并；必要时引导业务用 hash tag 拆分或冗余。
  - 热点是分布式 KV 的固有难点，本设计提供识别与缓解机制，但需业务侧配合 Key 设计。
- **倾斜防护：** 限制 hash tag 的滥用，避免把过多 Key 钉到同一槽位造成分片倾斜。

## 6. 元数据一致性与脑裂防护

- PD 自身 3 节点 Raft，元数据强一致、高可用。
- 每个分片携带 **epoch**；任何携带过期 epoch 的写/路由会被拒绝并触发路由刷新，
  防止迁移/分裂过程中向旧位置写入造成不一致。

## 7. 扩展性小结

| 增长来源 | 应对手段 |
| --- | --- |
| 在线人数上升 → 写 QPS 上升 | 加节点 + 分片再均衡/分裂，写吞吐线性扩展 |
| 数据量上升 | 加节点分担容量，SSD 主存成本可控 |
| 读 QPS 上升 | 加 Follower 副本 + 开启 Follower Read 扩展读 |
| 局部热点 | 热点识别 + Proxy 缓存/写合并 + Key 设计引导 |
