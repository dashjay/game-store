# 07 · 备份、恢复与数据迁移

> 本文件说明 GameStore 的备份/恢复策略、按时间点恢复（PITR），以及如何从现有 Redis 平滑迁入。

## 1. 备份目标

- **抗逻辑错误：** 误删、误改、版本发布事故等 —— 多副本无法防护（错误会被复制到所有副本），必须靠备份。
- **抗集群级灾难：** 整集群/整地域不可用时，可从对象存储重建。
- **可审计、可回溯：** 满足合规与"赛季前快照"等业务需求。

> 关键认知：**多副本（Raft）≠ 备份**。副本防硬件/节点故障，备份防逻辑错误与灾难。两者都需要。

## 2. 备份组成

### 2.1 全量快照（基于 RocksDB Checkpoint）
- 利用 RocksDB **Checkpoint** 在某一致点对 SST 文件做硬链接快照，几乎不阻塞前台写。
- 以 **分片** 为单位生成快照（含该分片状态机数据与对应的 Raft applied index），上传到对象存储。
- 由 Operator 按 `backup.schedule`（如每 6 小时）周期触发，跨分片快照点用各自 applied index 标定。

### 2.2 增量 / 日志归档（支撑 PITR）
- 持续把 **已提交的 Raft 日志**（或等价的变更流）归档到对象存储。
- 全量快照之后的所有变更都可由日志重放补齐，从而支持 **任意时间点恢复（PITR）**。

### 2.3 存放
- 统一存到对象存储（S3 / GCS / OSS），开启版本化与生命周期策略（如冷归档、过期清理）。
- 备份对象按 `集群 / 分片 / 时间点` 组织，便于检索与按分片并行恢复。

## 3. 恢复

### 3.1 全量恢复
1. 选定恢复目标（最近快照或指定快照）。
2. Operator 拉起新集群骨架，从对象存储 **按分片并行** 下载快照 SST 并 ingest 到各分片 Leader。
3. 各分片用快照重建状态机并通过 Raft 把数据复制给新副本，恢复复制因子。
4. PD 重建槽位映射，Proxy/SDK 感知新拓扑后对外提供服务。

### 3.2 按时间点恢复（PITR）
1. 加载目标时间点 **之前最近的全量快照**。
2. 从快照的 applied index 起，**重放归档的 Raft 日志** 到目标时间点（精确到秒级或更细）。
3. 适用于"回滚到事故发生前一刻"的场景。

### 3.3 单分片恢复
- 因备份以分片为单位组织，可只恢复受影响的分片，缩短 RTO，不必整集群回滚。

## 4. RPO / RTO 目标（建议）

| 指标 | 目标方向 | 决定因素 |
| --- | --- | --- |
| RPO（可容忍数据丢失窗口） | 趋近于 0（PITR 到秒级） | 日志归档频率与延迟 |
| RTO（恢复耗时） | 尽量短 | 快照大小、对象存储带宽、按分片并行度 |

> 具体数值需结合数据量与对象存储吞吐压测确定，并写入运维 SLO。

## 5. 从现有 Redis 迁移

业务此前用 Redis 当持久库，迁移需"协议兼容 + 数据搬运 + 平滑切换"三步：

### 5.1 协议兼容
- GameStore 兼容 RESP 与常用命令/类型，业务连接串/客户端基本无需改造（见 [`03-storage-engine.md`](03-storage-engine.md)）。

### 5.2 数据搬运
- **离线导入：** 解析 Redis RDB，按 GameStore 编码批量 ingest（适合停机窗口允许的场景）。
- **在线同步：** 用 `redis-shake` 类工具，先全量再订阅增量（`PSYNC`/`replicaof` 风格）持续同步到 GameStore，
  直至增量延迟趋零（适合不停服迁移）。

### 5.3 平滑切换
- **双写校验（可选）：** 迁移期对 Redis 与 GameStore 双写并比对，验证一致性与功能兼容。
- **灰度切流：** 按业务/分片逐步把读写流量从 Redis 切到 GameStore，保留快速回退路径。
- **下线 Redis：** 流量全切且观察期通过后，停用 Redis，释放内存成本。

## 6. 演练

- 备份/恢复必须 **定期演练**（如季度级），验证备份可用、RTO/RPO 达标。
- 未经恢复验证的备份视为不可信。
