# 04 · 复制、一致性与冲突解决（无主多写）

> 本文件是 GameStore 的核心机制文档。它说明系统如何用 **无主多写（Leaderless Multi-Write）+ 可调 Quorum +
> HLC/LWW/CRDT 冲突解决 + Anti-Entropy 修复** 同时满足 **极高可用**、**不丢数据** 与 **（可选的）强一致**。
>
> **路线说明：** 本文件在 MR-0007 由原"Multi-Raft（CP）"方案重写为"无主多写"方案，以对齐字节跳动 Abase。
> 详见 [`../EVOLUTION.md`](../EVOLUTION.md) MR-0007。

## 1. 为什么不用 Raft：选型论证

直觉上，"用 Raft 保证多副本一致"是分布式 KV 的常见做法（TiKV 即如此）。但若目标是 **极高可用**
（对齐 Abase 所服务的推荐/广告/信息流/游戏等场景），共识/主从协议存在两个难以回避的代价：

1. **选主/切主停顿。** 主从或单 Raft 组在 Leader 故障时需要心跳探测 + 重新选举，工程实践普遍是 **秒级不可写**。
   而业务要求 P99 在 10ms 量级、不可接受秒级抖动。
2. **慢节点问题。** 有主架构的写是单点：当主"没彻底挂但变慢"（盘热降速、CPU 负载高、网卡抖动）时，
   所有写都被拖慢，且难以靠阈值准确踢除；有主架构按 **日志 sequence 顺序同步**，前面卡住后面就卡住。

> 字节跳动 Abase2 的公开资料明确做了 2PC / Raft / Quorum 的对比，结论是：
> **"quorum 协议由于没有选主，可用性更高"**，因而选择 **无主多写**（参考 Dynamo 并结合多主架构优点）。
> 来源：火山引擎《字节跳动极高可用 KV 存储系统详解》。

因此 GameStore **不以 Raft 作为主复制机制**。Raft 作为"已评估并否决"的备选保留于此，其代价即否决理由。
（仍为对一致性要求高、可容忍偶发切换的业务保留 **单主半同步** 选项，见 §5.2。）

## 2. 复制模型：Partition + N Replica + Replica Coordinator

- 键空间被切成多个 **Partition（分片）**，每个 Partition 有 **N 个 Replica**（默认 N=3 或 5），跨 AZ/POD 放置。
- **无主：** Partition 的 **任一 Replica 都能接受读写**。处理某次请求的那个副本充当 **Replica Coordinator**。
- 只要 Partition **还有副本存活**，即可对外服务——这是"极高可用"的来源（无主可挂、无选主等待）。

```
Partition P (N=3，跨 3 AZ)
   Replica A (AZ1)  ── 可作为 Coordinator ──┐
   Replica B (AZ2)  ── 可作为 Coordinator ──┤  任一副本均可读写
   Replica C (AZ3)  ── 可作为 Coordinator ──┘
```

> 对照 Raft：这里没有"唯一 Leader"，也没有"选主"。副本对等，写入可乱序提交、不依赖前序数据是否同步完成。

## 3. 写路径与 Quorum

以 `HSET player:{id} gold 100`（W=2, N=3）为例：

1. Proxy/SDK 按元信息路由到该 Partition 的 **某个就近副本**（设为 Replica B），由其 Coordinator 处理。
2. Coordinator 为该写 **分配全局唯一的 HLC 时间戳**（作为版本号与全排序依据，见 §6）。
3. 写入先落本地 **WAL**（保证持久化），同时 **并发 forward 到其余副本**。
4. 当收到 **≥ W 个副本的 WAL 落盘成功响应**（可不含本地）即向客户端返回成功；否则在超时内失败。
5. 落盘后的数据提交到引擎 **数据暂存层**；达到条件后合并下刷到 **通用引擎层**，对应 WAL 即可 GC（见 [`03-storage-engine.md`](03-storage-engine.md)）。

> **不丢数据：** 成功 ⇒ 至少 W 个副本已 WAL 落盘。`W≥2` 时，任意单副本（单 AZ）故障，数据仍在其余副本。
> **极致写可用：** `W=1` 时只要任一副本可写即成功（缓存场景），其余副本异步补齐。

### 3.1 高频写优化（与负载模型协同）
- **每 Core 一个共享 WAL：** 一个 Core 上的多个 Replica 共享一个 WAL，**合并不同 Replica 的碎片化提交**，减少 IO 次数。
- **Run-to-Complete 协程：** 每个 Core 单线程协程框架，请求从处理到返回不切线程，避免上下文切换开销。
- **CRDT 计数器天然契合：** "每字段 1~2 次/秒"的 `INCR`/`HINCRBY` 在多写下并发合并而不冲突（见 §7），无需串行化到单点。

## 4. 读路径

1. Proxy/SDK 按元信息 + **地理位置** 把读路由到合适的副本。
2. 该副本 Coordinator 依 **一致性策略** 查询本地（或按需远端）存储，
   将结果按 **冲突解决规则合并**（暂存层多版本 + 引擎层单版本，见 [`03-storage-engine.md`](03-storage-engine.md)）后返回。
3. 读慢节点可用 **Backup Request**：某副本超过 P99/80 分位未返回时，向其他副本补发请求取最快结果。

- **R=1（默认，最终一致）：** 读最近的单副本，最快；可能读到尚未收敛的旧值，但正常网络下秒级内即最终一致。
- **R 增大 / `W+R>N`：** 读多个副本并按 HLC 取最新合并，获得更强一致（见 §5）。

## 5. 一致性级别（可调旋钮）

GameStore 把一致性做成 **业务可配置的旋钮**（按 Namespace/Table/Key 粒度），而非全局定死：

### 5.1 无主多写 + Quorum（默认）
- 配置 `N / W / R`。`W`/`R` 越大越偏一致与可靠，越小越偏可用与低延迟。
- **`W+R>N`（典型 W=3,R=3,N=5）：** 读写集合必然相交，可获得 **无更新场景下的写后读一致**——
  既"写成功的数据立即能读到"，又 **没有主从切换的秒级不可用**。适合多数"要立即可见但不能抖动"的业务。
- **`W=1,R=1`：** 极致可用/低延迟，最终一致，适合缓存类、可容忍偶发旧读的场景。

### 5.2 单主半同步（Leader&Followers，可选强一致）
- 选出唯一主副本处理读写，**至少 2 个副本同步落盘后才确认**（类 MySQL semi-sync），提供线性一致；
  单点故障时新主仍有全量数据。
- 代价：接受故障时的 **短时切换不可用**——这是用一点可用性换强一致，供对一致性要求高的业务选择。

### 5.3 给"玩家财产/金币"数据的建议
- **高频统计/状态字段**（在线时长、HP、坐标等）：多主 + `W=1/2` + **CRDT 计数器**，吞吐与可用性最佳。
- **金币/钻石/财产等强一致字段**：用 **`W+R>N` 写后读一致** 或 **单主半同步**，确保不读到旧余额；
  并优先用 **CRDT 计数器（`INCRBY`）** 表达增减，避免"读-改-写"在多写下丢更新。

## 6. 冲突解决：HLC + LWW

多点写入必然产生冲突（同一 Key 在不同副本被并发修改）。解决方案：

- **版本化：** 每次写入用 **HLC（Hybrid Logical Clock）时间戳** 作为全局唯一、可比较的版本号。
  HLC 结合物理时钟与逻辑计数 + 副本位置信息，保证"有交互的后写"时间戳一定更大，且全球唯一、可比较。
- **LWW（Last Write Wins）：** 对幂等命令（如 `SET`），冲突时保留 HLC 最大的版本。简单且对多数业务足够。
- 对 **非幂等命令** 与复杂结构，LWW 不足以保语义，引入 CRDT（见 §7）。

## 7. CRDT：兼容 Redis 非幂等命令与复杂结构

`INCR`/`APPEND` 与 Hash/ZSet/List 等若用 LWW 会丢失并发更新（如两次 `INCR` 互相覆盖）。
GameStore 采用 **Operation-based（op-based）CRDT**，**完全兼容 Redis 语义**：

- 每个操作分配全球唯一 **HLC 时间戳** 作为全排序依据；记录到 **Operation 日志（ReplicaLog）**。
- 某个 Key 的最终值 = 其所有 Operation 按时间戳排序后合并的结果。**只要副本间 Operation 日志一致，最终状态必然一致。**
- 为避免日志膨胀与查询时反复合并的开销：**定期 Checkpoint**，把已达成一致时间戳之前的操作合并为单一结果存入通用引擎层，
  并用内存 Cache 缓存最终值，使查询走点查、不必访问 Operation 日志。
- 已实现/规划覆盖 **String / Hash / ZSet / List** 的 CRDT，保持与 Redis 一致的命令与返回语义。

## 8. Anti-Entropy：高效的一致性检测与修复

当 `W<N` 时，部分副本先返回成功，其余副本可能因网络抖动暂未收到数据。**Anti-Entropy** 异步补齐：

- 在 WAL 之上构建 **ReplicaLog 索引**：每个副本负责一条 ReplicaLog，写入分配 **严格递增的 Seqno**，
  并在其他副本上保有该 ReplicaLog 的副本。
- 各副本后台定期：
  1. 取自身 ReplicaLog 进度向量 `[Seqno1, …, SeqnoN]`；
  2. 与其他副本交换进度向量；
  3. 比对，发现自身某条 ReplicaLog 落后则向对应副本拉取缺失数据并提交引擎；
  4. 当某条 ReplicaLog 在 SeqnoX 前已全副本一致，回收 SeqnoX 前的日志。
- **相比 Dynamo/Cassandra 的 Merkle-tree 全量 Diff**：ReplicaLog 通常只需常驻内存的少量进度向量即可完成检测，
  正常网络下 **秒级收敛**，避免读 QPS 放大与长修复周期（极端如网络分区时才 dump 到磁盘）。

## 9. 故障与恢复

- **副本变慢/短暂不可达：** 无主架构下写自动落到其他副本、读用 Backup Request 规避，**对用户基本无感**，无需切主。
- **副本宕机：** 不影响 Partition 可用性（只要还有副本）；恢复后由 Anti-Entropy 追平；
  长期失效则由 **MetaServer** 检测并在健康节点 **重建副本** 以恢复副本数（见 [`02-architecture.md`](02-architecture.md)）。
- **数据恢复期可用性：** 允许在主从落后过多时 **乱序提交**，恢复中也能尽快接受写入，兼顾一致性算法效率与可用性。

## 10. 与 Redis 主从 / Raft 的对比

| 维度 | Redis 主从+哨兵 | Raft（如 TiKV） | GameStore（无主多写，默认） |
| --- | --- | --- | --- |
| 写入点 | 单主 | 单 Leader | 任一副本 |
| 故障切换 | 秒级选主、可能丢已确认写 | 秒级选主、不丢已提交 | **无选主、无切换停顿** |
| 慢节点 | 拖慢全部写 | 受多数派中较慢者影响 | **架构上规避**（写他副本/Backup Request） |
| 一致性 | 异步最终一致 | 线性一致(CP) | 默认最终一致；`W+R>N`/单主半同步可强一致 |
| 持久性 | 主挂可能丢 | 多数派落盘 | **W 个副本 WAL 落盘**才确认 |
| 多写冲突 | —（单主无冲突） | —（单主无冲突） | HLC + LWW + **CRDT** 解决 |
| 多地域 | 中间件同步，难保最终一致 | 跨地域延迟高 | 多主天然支持，Main Replicator 主导同步 |

**结论：** GameStore 以"无主多写 + Quorum + CRDT"换取 Abase 式 **极高可用**，
同时用 **W 个副本 WAL 落盘 + 可选强一致模式** 守住"不丢数据"与"财产强一致"的底线。
