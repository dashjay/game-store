# 01 · 工作负载分析与数据模型

> 本文件把业务描述转化为可量化的负载模型，并据此给出 **数据模型（Key 设计）** 建议。
> 后续的架构、存储引擎、分片、容量规划都以本文的数字为共同前提。

## 1. 业务负载描述

- 存储对象：玩家的 **数据与财产**（在线状态、金币/钻石、背包、计数器、等级、buff 等）。
- 规模特征：**单玩家约 50 个字段**，**每个字段每秒被修改 1~2 次**。
- 值特征：以 **小值** 为主——整数计数器、短字符串、序列化后的小对象。
- 访问特征：登录时批量读取玩家状态；在线期间高频写少量字段；登出/落地时整体落盘。
- 持久性要求：财产类字段 **不可丢失**，需 **单 Key 线性一致**。

## 2. 量化建模

以单玩家为单位推导，便于按在线人数线性放大。

**单玩家写入速率：**

```
50 字段 × (1~2 次/秒) = 50 ~ 100 写操作/秒/玩家
```

**按在线人数放大（写 QPS）：**

| 同时在线玩家数 | 写 QPS（下限 50/人） | 写 QPS（上限 100/人） |
| --- | --- | --- |
| 10,000 | 0.5 M/s | 1 M/s |
| 100,000 | 5 M/s | 10 M/s |
| 1,000,000 | 50 M/s | 100 M/s |

**关键结论：写吞吐随在线人数线性增长，必然超出单机能力，因此 _水平分片_ 是硬需求**（见 [`05-sharding-routing.md`](05-sharding-routing.md)）。

**容量估算（粗略）：**

设单字段（键名 + 值 + 编码开销）平均 100 B，单玩家 50 字段 ≈ 5 KB（不含多版本与索引开销）。

```
1,000,000 注册玩家 × 5 KB ≈ 5 GB 逻辑数据
```

逻辑数据量本身不大，**真正的压力来自写入速率与由此产生的 LSM 写放大、Compaction 与多版本 GC**，
而非总容量。这进一步印证了"以写优化为核心"的设计方向（见 [`03-storage-engine.md`](03-storage-engine.md)）。

> 注意：上表是"裸写入"估算。若不加治理，50~100 次/秒/玩家会直接打到存储层。
> 下一节的数据模型与写入合并策略正是为降低这一放大而设计。

## 3. 数据模型设计

### 3.1 推荐：用 Hash 聚合单个玩家的字段

把一个玩家的 ~50 个字段聚合到 **一个 Hash Key** 下：

```
Key:   player:{player_id}
Type:  Hash
Field: gold, diamond, level, exp, hp, pos_x, pos_y, ...   (~50 个)
```

这样设计的收益：

1. **同分片局部性。** 一个玩家的所有字段落在同一个槽位 → 同一分片 → 同一 Raft 组，
   单玩家的读写不产生跨分片操作，天然满足"单玩家强一致"。
2. **Key 数量大幅下降。** 从"玩家数 × 50"个 String Key 降到"玩家数"个 Hash Key，
   减少元数据与路由开销。
3. **便于批量写合并。** 多个字段的更新可用单条 `HSET key f1 v1 f2 v2 ...` 或 Pipeline 合并，
   把"50~100 次/秒/玩家"的离散写聚合成少量批量写，显著降低 Raft 提交次数与写放大。

### 3.2 槽位归属与 hash tag

哈希分片按 **Key 的 CRC16 → 槽位** 计算。`player:{id}` 整体作为 Key 计算槽位即可保证单玩家同分片。
如需把"同一公会/房间"的多个玩家强制放到同一分片以便（未来的）跨玩家原子操作，可用 **hash tag**：

```
player:{guild_42}:1001     # 仅花括号内的 guild_42 参与槽位计算
player:{guild_42}:1002     # 与上一个落到同一槽位/分片
```

> 注意：过度使用 hash tag 会导致数据倾斜与热点分片，需谨慎，仅在确有跨 Key 原子需求时使用。

### 3.3 写入合并策略

为把"每字段 1~2 次/秒"的高频离散写转化为对存储友好的批量写，提供两层合并：

- **客户端/SDK 侧：** 鼓励用 `HSET` 一次写多个字段、用 Pipeline 批量提交。
- **Proxy 侧（可选）：** 在极小时间窗（如 1~5 ms）内对同一 Key 的多次字段更新做 **写合并（coalescing）**，
  合并为一次后端写。该策略需业务确认可接受 ms 级的写可见延迟（对游戏状态通常可接受）。

> 写合并是"性能 vs 实时可见性"的取舍，**默认关闭**，按业务显式开启并在 EVOLUTION 记录决策。

### 3.4 过期与清理

- 在线态、临时 buff 等可设置 **TTL**（`EXPIRE` / `PEXPIRE`），由元数据中的过期时间戳 + Compaction Filter 后台回收（见 [`03-storage-engine.md`](03-storage-engine.md)）。
- 财产类核心字段不设 TTL，长期持久保存。

## 4. 读写路径概览（与本模型相关的部分）

- **登录读取：** `HGETALL player:{id}` 或按需 `HMGET`，单分片一次往返即可拉全。
- **在线写入：** 高频 `HSET player:{id} field value`（建议多字段合并），经 Leader → 多数派落盘确认。
- **登出落地：** 业务可做一次完整 `HSET` 兜底，确保最终状态持久化。

## 5. 对后续设计的约束（Checklist）

- [x] 必须支持水平分片（写吞吐随在线人数线性增长）。
- [x] 必须对 **Hash 类型** 做一等公民优化（玩家数据的主载体）。
- [x] 存储引擎必须 **写优化**，并能高效处理同 Key 的高频覆盖与多版本 GC。
- [x] 必须支持单 Key 线性一致写，且写入持久化后才确认。
- [x] 应提供批量/合并写能力以降低 Raft 提交与写放大。
