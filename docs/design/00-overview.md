# 00 · 设计总览

> 本文件是 GameStore 设计文档体系的入口，统一陈述 **目标、非目标、整体取舍** 与 **设计原则**，
> 后续各文档在此前提下展开。阅读顺序建议按文件编号从小到大。

## 1. 背景与问题

公司过去把 Redis 作为"持久化数据库"来存储玩家数据与财产信息。这一选择在两方面持续付出代价：

- **数据不稳定。** Redis 以内存为主，持久化（RDB 快照 / AOF 日志）是补充手段。
  在宕机、OOM、主从切换、AOF 重写等场景下，存在丢失"最后一段写入"的窗口。
  对"玩家财产"这种 **不可丢失、需强一致** 的数据，这是不可接受的风险。
- **成本居高不下。** 全内存模型要求把全量数据常驻 RAM。随着玩家数与单玩家数据量增长，
  内存（及为冗余而翻倍的内存）成本线性甚至超线性上升。

业界已有成熟的解法范式：用 **磁盘（SSD）做主存储 + 多副本一致性协议保证可靠 + 兼容 Redis 协议降低迁移成本**。
代表系统：

- **TiKV** —— Multi-Raft + RocksDB + Placement Driver，分布式事务型 KV。
- **Apache Kvrocks** —— 在 RocksDB 之上实现 Redis 协议与数据结构的磁盘型 KV。
- **Abase（字节跳动）** —— 兼容 Redis 协议的高可用持久化 KV 存储，强调多副本与运维自动化。

GameStore 综合上述范式，针对 **高频小值更新的游戏状态** 场景做定制。

## 2. 设计目标（Goals）

1. **强持久化、不丢数据。** 写入须经 Raft 多数派持久化到磁盘后才向客户端确认成功。
2. **兼容 Redis 协议。** 支持 RESP2/RESP3 与常用数据类型，业务以最小改动迁移。
3. **高可用。** 单节点或单可用区故障不影响整体可用性，故障由系统自动恢复。
4. **水平扩展。** 容量与吞吐随节点数近似线性增长，扩缩容对业务透明。
5. **低成本。** 以 SSD 为主存、内存仅作缓存，存储成本随数据量平稳增长。
6. **云原生易部署。** 在主流公有云上可一键部署、跨 AZ、自动化运维。
7. **写优化。** 针对"每玩家约 50 字段、每字段 1~2 次/秒"的高频小写入做专门优化。

## 3. 非目标（Non-Goals，至少初期不做）

- **完整复刻 Redis 全部命令。** 优先支持高频核心命令与类型；Pub/Sub、Lua、Stream 等按需在后期补齐。
- **跨分片分布式事务。** 初期只保证 **单 Key / 单分片** 的线性一致；跨分片事务列为后期可选项。
- **多地域 Active-Active。** 初期面向单地域多可用区；跨地域多活后期评估。
- **作为分析型（OLAP）存储。** GameStore 面向在线点查/点写（OLTP-like），不承担大范围扫描分析。

## 4. 整体取舍（Key Trade-offs）

| 维度 | 选择 | 取舍理由 |
| --- | --- | --- |
| 主存介质 | SSD（RocksDB/LSM） | 成本远低于 RAM；LSM 对写入友好，契合高频写场景 |
| 可靠性 | Raft 多数派落盘 | 用一致性协议换"不丢数据"，比 Redis 异步持久化强 |
| 一致性范围 | 分片内线性一致 | 满足玩家财产强一致需求，同时避免全局事务的复杂度与开销 |
| 接入协议 | Redis RESP | 复用成熟生态与客户端，业务零/低改造迁移 |
| 分片方式 | 哈希槽（兼容 Redis Cluster） | 负载均匀、路由简单、与 Redis 生态一致 |
| 拓扑 | Proxy + 存储节点 + PD | 解耦路由/存储/调度，便于扩缩容与多语言客户端接入 |

## 5. CAP 立场

GameStore 在 **单个分片内是 CP 系统**：分片由一个 Raft 组承载，写入需多数派确认。
当某分片丢失多数派（如同时挂掉 2/3 副本）时，**该分片** 暂时不可写（牺牲可用性以保证一致性与不丢数据），
直至多数派恢复或 PD 完成副本重建。而 **其他分片不受影响**，整体集群仍对外提供服务。

可选的 **Follower Read**（从副本读取）在显式开启时提供有界陈旧（bounded staleness）的读，
用于扩展读吞吐——这是一个由业务按 Key/命令粒度选择的、明确的一致性让步。

## 6. 设计原则

- **可靠性优先于性能；性能优先于功能丰富度。** 先把"不丢数据 + 高可用"做扎实，再谈吞吐与命令覆盖。
- **以真实负载为准绳。** 所有容量、吞吐、调参决策都应回到 [`01-workload-data-model.md`](01-workload-data-model.md) 的负载模型。
- **机制与策略分离。** 例如：副本放置（机制在存储节点）与放置决策（策略在 PD）解耦。
- **可观测、可运维、可回滚。** 没有指标的功能视为未完成；每个上线项都要能灰度与回退。
- **渐进式落地。** 见 [`09-roadmap.md`](09-roadmap.md)，从单机 MVP 起步，逐步演进到完整分布式形态。

## 7. 术语表

| 术语 | 含义 |
| --- | --- |
| Node / 存储节点 | 运行存储引擎与多个 Raft 副本的进程/实例 |
| Shard / 分片 | 键空间的一个子集，由一个 Raft 组负责复制 |
| Replica / 副本 | 某分片在某节点上的一份数据拷贝（Leader 或 Follower） |
| Raft Group | 负责一个分片复制的 Raft 共识组（默认 3 副本） |
| Slot / 槽位 | 哈希分片的最小单位，共 16384 个，多个槽位归属一个分片 |
| PD（Placement Driver） | 集群元数据中心与调度器，管理拓扑、槽位映射、均衡与故障转移 |
| Proxy | 无状态接入层，解析 RESP、按槽位路由、连接复用与热点缓解 |
| Leader Lease | Leader 持有的时间租约，用于在租约内提供无需多数派往返的本地一致读 |
| ReadIndex | 一种线性一致读机制，读前确认仍是 Leader 且已应用到读点 |
