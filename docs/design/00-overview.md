# 00 · 设计总览

> 本文件是 GameStore 设计文档体系的入口，统一陈述 **目标、非目标、整体取舍** 与 **设计原则**，
> 后续各文档在此前提下展开。阅读顺序建议按文件编号从小到大。

## 1. 背景与问题

公司过去把 Redis 作为"持久化数据库"来存储玩家数据与财产信息。这一选择在两方面持续付出代价：

- **数据不稳定。** Redis 以内存为主，持久化（RDB 快照 / AOF 日志）是补充手段。
  在宕机、OOM、主从切换、AOF 重写等场景下，存在丢失"最后一段写入"的窗口。
  对"玩家财产"这种 **不可丢失、需强一致** 的数据，这是不可接受的风险。
- **成本居高不下。** 全内存模型要求把全量数据常驻 RAM。随着玩家数与单玩家数据量增长，
  内存（及为冗余而翻倍的内存）成本线性甚至超线性上升。

业界已有成熟的解法范式：用 **磁盘（SSD）做主存储 + 多副本一致性协议保证可靠 + 兼容 Redis 协议降低迁移成本**。
代表系统：

- **Abase / Abase2（字节跳动，主要对标）** —— 完全兼容 Redis 协议的高可用持久化 KV 存储。
  其第二代为追求"极高可用"而 **刻意不使用 Raft/主从共识**，采用 **无主多写（Leaderless Multi-Write）+ 可调 Quorum（W+R>N）+
  HLC/LWW/Operation-based CRDT + Anti-Entropy**，默认最终一致，并提供单主半同步等强一致可选模式。
- **Apache Kvrocks** —— 在 RocksDB 之上实现 Redis 协议与数据结构的磁盘型 KV（本项目主要借鉴其单机编码层）。
- **TiKV** —— Multi-Raft + RocksDB + PD 的分布式事务型 KV（**作为对照**：它走的是 CP/共识路线，
  与 Abase 的 AP/无主路线相反；本项目经评估后选择对齐 Abase，理由见 [`04-replication-consistency.md`](04-replication-consistency.md) §1 与 [`../EVOLUTION.md`](../EVOLUTION.md) MR-0007）。

GameStore 以 **Abase 式无主多写** 为骨架，结合 **Kvrocks 式 Redis-on-RocksDB 编码**，
针对 **高频小值更新的游戏状态** 场景做定制。

> **重要路线说明：** 本文档体系第一版（MR-0001~0006）曾以 Multi-Raft 为复制机制。
> 经查证 Abase 并不使用 Raft，且本项目目标是做"与 Abase 很接近的产品"，故于 **MR-0007** 修正为无主多写路线。
> 凡涉及"复制/一致性/高可用"的设计，以本次修正后的描述为准。

## 2. 设计目标（Goals）

1. **强持久化、不丢数据。** 写入须达到 **Quorum**（用户配置的 W 个副本完成 WAL 落盘）后才确认；
   对强一致需求可选 `W+R>N` 写后读一致或单主半同步模式。
2. **兼容 Redis 协议。** 支持 RESP2/RESP3 与常用数据类型；非幂等命令与复杂结构用 **CRDT** 保证语义完全兼容。
   **默认用标准 Redis 客户端直连 Proxy，无需自研 SDK**（重型 SDK 仅为可选优化，见 [`02-architecture.md`](02-architecture.md) §3.1）。
3. **极高可用。** 对齐 Abase：**消除选主/主从切换的秒级不可用、从架构上规避慢节点**；单节点或单可用区故障不影响整体可用性。
4. **可调一致性。** 默认最终一致（无主多写），向用户开放 `Quorum(W/R/N)` 与"多主/单主"模式选择，按业务取舍。
5. **水平扩展。** 容量与吞吐随节点数近似线性增长，扩缩容对业务透明。
6. **低成本。** 以 SSD 为主存、内存仅作缓存与暂存层，存储成本随数据量平稳增长；多租户资源池化进一步降本。
7. **云原生易部署。** 在主流公有云上可一键部署、跨 AZ、自动化运维。
8. **写优化。** 针对"每玩家约 50 字段、每字段 1~2 次/秒"的高频小写入做专门优化（CRDT 计数器天然契合）。

## 3. 非目标（Non-Goals，至少初期不做）

- **完整复刻 Redis 全部命令。** 优先支持高频核心命令与类型；Pub/Sub、Lua、Stream 等按需在后期补齐。
- **默认强一致 / 全局事务。** 默认走最终一致；强一致是 **按表/按 Key 可选** 的能力，而非默认。跨分片事务列为后期可选项。
- **作为分析型（OLAP）存储。** GameStore 面向在线点查/点写（OLTP-like），不承担大范围扫描分析。

## 4. 整体取舍（Key Trade-offs）

| 维度 | 选择 | 取舍理由 |
| --- | --- | --- |
| 主存介质 | SSD（RocksDB/LSM） | 成本远低于 RAM；LSM 对写入友好，契合高频写场景 |
| 复制机制 | **无主多写 + 可调 Quorum**（非 Raft） | 消除选主停顿、规避慢节点，获得极高可用；不丢数据靠 W 个副本 WAL 落盘 |
| 一致性 | 默认最终一致，可调强一致 | 可用性优先；强一致按表/Key 选择（`W+R>N` 或单主半同步），满足财产类需求 |
| 冲突解决 | HLC + LWW + Operation-based CRDT | 多写下保证最终一致且 **完全兼容 Redis 语义**（含 `INCR`/`APPEND`） |
| 接入协议 | Redis RESP | 复用成熟生态与客户端，业务零/低改造迁移 |
| 分片方式 | 哈希到 Partition | 负载均匀、路由简单；可在 Proxy 层映射为 Redis Cluster 槽位 |
| 拓扑 | Proxy/重型 SDK + DataNode + MetaServer/RootServer | 解耦路由/存储/调度，便于扩缩容、多租户与多语言接入 |

## 5. CAP 立场

GameStore 与 Abase 一致，**偏向 AP**：在可用性与一致性之间 **默认优先可用性**，并把强一致作为 **可配置能力** 开放给业务。

- **默认（多主/无主，最终一致）：** 分片的 **任一副本均可读写**，只要有一个副本存活即可服务，
  写入达到 Quorum `W` 即确认；副本间通过 **Anti-Entropy** 在秒级（正常网络下）收敛，冲突由 **HLC + LWW/CRDT** 解决。
  这消除了选主停顿与慢节点对写入的影响——即"极高可用"。
- **强一致（可选）：** 业务可按表/Key 配置：
  - **`W+R>N` Quorum**（典型 W=3,R=3,N=5）获得无更新场景下的 **写后读一致**；
  - **单主半同步（Leader&Followers）**：唯一主处理读写、至少 2 副本同步后才确认，提供线性一致，
    代价是接受单点故障时的短时切换（类 MySQL semi-sync）。
- **CAP 不是非黑即白：** 我们不被 CA 二选一束缚，而是把 N/W/R 与主从模式做成 **旋钮**，让业务自行落点。

## 6. 设计原则

- **可用性优先；可靠性（不丢数据）通过 Quorum 保证；其后才是性能与功能丰富度。** 这与 Abase 的取舍一致。
- **以真实负载为准绳。** 所有容量、吞吐、调参决策都应回到 [`01-workload-data-model.md`](01-workload-data-model.md) 的负载模型。
- **机制与策略分离。** 例如：副本放置（机制在 DataNode）与放置决策（策略在 MetaServer）解耦。
- **可观测、可运维、可回滚。** 没有指标的功能视为未完成；每个上线项都要能灰度与回退。
- **渐进式落地。** 见 [`09-roadmap.md`](09-roadmap.md)，从单机 MVP 起步，逐步演进到完整分布式形态。

## 7. 术语表

| 术语 | 含义 |
| --- | --- |
| DataNode | 数据存储节点（线上每块盘一个），内含多个 Core |
| Core | DataNode 的最小资源单位（绑定一个 CPU 核），Run-to-Complete 协程框架，承载多个 Replica |
| Partition / 分片 | 键空间的一个子集（一个 Table 切成多个 Partition），由 N 个 Replica 复制 |
| Replica / 副本 | 某 Partition 的一份数据拷贝；无主模式下任一副本都可读写 |
| Replica Coordinator | 接收某次读写的副本上的协调者：分配 HLC、并发转发到各副本、按 Quorum 判定成功 |
| Quorum (N/W/R) | 副本数 N、写成功所需最少响应 W、读所需最少响应 R；`W+R>N` 可得写后读一致 |
| HLC | Hybrid Logical Clock，全局唯一可比较时间戳，用作数据版本号与全排序依据 |
| CRDT | 无冲突复制数据类型；本系统用 Operation-based CRDT 兼容 Redis 非幂等命令与复杂结构 |
| Anti-Entropy | 基于 ReplicaLog(Seqno) 进度向量的异步一致性检测与修复机制 |
| MetaServer | 元信息中心 + 多租户 QoS 总控 + 故障检测/数据修复调度 |
| RootServer | 全集群视角的轻量协调组件，负责跨集群资源/迁移/容灾 |
| Slot / 槽位 | （可选）Proxy 层为兼容 Redis Cluster 而暴露的 16384 槽位映射 |
| Proxy | 无状态接入层（RESP2/Thrift），按元信息路由、连接复用、重试/Backup Request/热点与限流 |
| 重型 SDK | 跳过 Proxy 直连 DataNode 的客户端封装，面向延迟敏感的重度用户 |
| DTS | Data Transfer Service，负责迁移/备份回滚/订阅等数据流转 |
